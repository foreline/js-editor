# Code Block — Root Cause Analysis & Refactoring Plan

## Status
Accepted

## Summary

This document provides a detailed root-cause analysis of all six reported Code Block issues, along with a recommended fix strategy. The conclusion is to **refactor** (not rewrite) the existing `CodeBlock` class, as the class structure and contract are sound — the bugs stem from specific oversights in four areas: trigger stripping, DOM content scoping, toolbar toggling, and backspace handling.

## Issues and Root Causes

### Issue 1: Code block created by triple backticks contains unexpected text

**Symptom:** After typing `` ``` `` and pressing Enter, the code block contains leftover backtick characters and/or the language selector option text (`Plain textJavaScriptTypeScriptPythonJavaC#PHPCSSHTMLJSONSQLBash`).

**Root Cause:** Two separate bugs converge here.

1. **Keybuffer path skips trigger stripping.** In `KeyHandler.handleEnterKey()` (lines 135–157), the triple-backtick detection creates a `CodeBlock` and calls `applyTransformation(currentBlock, editor)` directly — bypassing `Editor.convertBlockType()`, which is the method that strips the markdown trigger from the element's `textContent`. As a result, `applyTransformation` reads `targetElement.textContent` (which still contains `` ``` ``), and sets `code.textContent = "` `` "``.

2. **Language selector text leaks into `textContent`.** `applyTransformation` appends a `<div class="language-selector"><select>…</select></div>` to the block. The `<select>` element's `<option>` children have display text (`Plain text`, `JavaScript`, etc.), so `blockElement.textContent` returns the concatenation of all visible text — including those option labels. This phantom text also affects Issue 5 (backspace).

**Fix:**
- In `KeyHandler.handleEnterKey()`, strip the backtick trigger from `currentBlock.textContent` **before** calling `applyTransformation`, aligning with the `convertBlockType` flow.
- Alternatively, make the keybuffer path delegate to `Editor.convertBlockType()` instead of calling `applyTransformation` directly.

### Issue 2: Code highlighting not working after triple-backtick creation

**Symptom:** A code block created via `` ``` `` has no syntax highlighting, even after selecting a language.

**Root Cause:** `applyTransformation()` sets `code.textContent = existingContent` but never calls `highlightSyntax()`. Compare this with `renderToElement()`, which does apply highlighting. Additionally, no language is set during backtick creation, so highlighting has nothing to work with initially — but even after selecting a language with the dropdown, `refreshHighlighting()` has a scoping bug (see Issue 4).

**Fix:** After setting `code.textContent`, call `this.highlightSyntax()` and set `code.innerHTML = highlighted` (only when content is non-empty and a language is known). For the initial case (no language), this is expected to be plain text, so no change is needed — the real highlight should happen on language selection (see Issue 4 fix).

---

### Issue 3: `refreshHighlighting()` uses global selector — only works for first code block

**Symptom:** Changing the language on any code block other than the first one in the document has no effect.

**Root Cause:** `refreshHighlighting()` in `CodeBlock.js` (line 270) uses:
```javascript
const element = document.querySelector(`[data-block-type="code"]`);
```
This always selects the **first** code block in the DOM, not the one associated with `this` instance. The `CodeBlock` instance needs a reference to its own DOM element.

**Fix:** Use `this._element` (inherited from `BaseBlock`) instead of a global `document.querySelector`. The element reference is set by `Editor.convertBlockType()` via `newBlock.element = blockElement`. Ensure `applyTransformation` also stores the element reference.

---

### Issue 4: Toolbar button state not updating after code block transformations

**Symptom:** When a code block is active, formatting buttons (bold, italic, underline, etc.) remain enabled. The code button itself doesn't show an "active" state.

**Root Cause:** `CodeBlock` does not override `getDisabledButtons()` — it inherits the base class default (`[]`). The `updateToolbarButtonStates()` method in `Editor.js` queries `getDisabledButtons()` to determine which buttons to disable, but gets an empty array. Formatting actions (bold, italic, strikethrough, underline) make no sense inside a `<code>` element and should be disabled.

**Fix:** Override `getDisabledButtons()` in `CodeBlock` to return the list of formatting button classes that should be disabled when a code block is active.

---

### Issue 5: Users cannot remove empty code block with Backspace

**Symptom:** Pressing Backspace at the beginning of an empty code block does nothing.

**Root Cause:** Two interrelated problems:

1. **Empty-check fails due to language selector text.** `KeyHandler.handleBackspaceKey()` checks `Utils.stripTags(currentBlock.innerHTML).trim()`. Because the code block DOM contains a `<select>` with `<option>` children, `stripTags` returns `"Plain textJavaScriptTypeScriptPythonJavaC#PHPCSSHTMLJSONSQLBash"`, which is never `=== ''`. The block always appears non-empty.

2. **No block-level backspace handler.** `CodeBlock` does not implement `handleBackspaceKey()`. Even if the empty-check were corrected, there's no logic to convert a code block back to a paragraph when backspace is pressed at position 0. Other block types (like heading blocks) handle this by converting back to paragraph.

**Fix:**
- Override `handleBackspaceKey()` in `CodeBlock`: when the `<code>` element is empty and the cursor is at position 0, convert the block back to a paragraph.
- Alternatively, fix the empty-check in `KeyHandler` to look at `code.textContent` (the inner editable element) rather than `currentBlock.innerHTML` for code blocks.

---

### Issue 6: Toolbar button click does not toggle code block state

**Symptom:** Clicking the code toolbar button when already inside a code block creates a new block instead of converting back to paragraph.

**Root Cause:** `Editor.convertCurrentBlockOrCreate()` only converts paragraph blocks. For any non-paragraph block, it falls through to `createNewBlock()`:
```javascript
if (currentBlockType === BlockType.PARAGRAPH) {
    return this.convertBlockType(currentBlock, targetBlockType, triggerText);
}
// For non-paragraph blocks, create a new block after current
return this.createNewBlock(targetBlockType, options);
```
There is no toggle logic: "if current block is already the target type, convert back to paragraph."

**Fix:** Add toggle detection at the top of `convertCurrentBlockOrCreate()`: if `currentBlockType === targetBlockType`, convert the block back to paragraph instead. This would also benefit other block types (headings, lists, etc.).

---

## Decision: Refactor, Not Rewrite

The `CodeBlock` class structure is fundamentally sound:
- Clean inheritance from `BaseBlock`
- Correct DOM structure (`pre > code` + language selector)
- Working serialization (`toMarkdown`, `toHtml`, `parseFromMarkdown`, `parseFromHtml`)
- Proper Prism.js integration via `SyntaxHighlighter`

The bugs are **integration-level issues** — mismatches between `CodeBlock` and the `Editor`/`KeyHandler`/`Toolbar` contract. A rewrite would not fix these; they require targeted corrections at the integration points.

## Implementation Plan

### Phase 1: Core Bug Fixes (CodeBlock.js)

- [x] **1.1** Override `getDisabledButtons()` — disable formatting buttons (bold, italic, underline, strikethrough, inline-code) for code blocks
- [x] **1.2** Fix `refreshHighlighting()` — use `this._element` instead of `document.querySelector`
- [x] **1.3** Fix `applyTransformation()` — store `this._element` reference, clear backtick content properly
- [x] **1.4** Implement `handleBackspaceKey()` — convert empty code block back to paragraph on Backspace at position 0

### Phase 2: Integration Fixes (Editor.js, KeyHandler.js)

- [x] **2.1** Fix `KeyHandler.handleEnterKey()` — guard keybuffer code block conversion to only run when the block is still a paragraph, preventing double-conversion with the input event listener
- [x] **2.2** Fix `Editor.convertCurrentBlockOrCreate()` — add toggle logic (code → paragraph) when current block already matches target type
- [x] **2.3** Fix `KeyHandler.handleBackspaceKey()` — for code blocks, check `<code>` element text instead of outer `innerHTML`
- [x] **2.4** Fix `CodeBlock.handleEnterKey()` — return `true` to prevent the default "add new block at end" behavior; let the browser insert a newline
- [x] **2.5** Add Ctrl/Cmd+Enter handling — when pressed inside a code block, create a new paragraph block below

### Phase 3: Tests

- [ ] **3.1** Update `tests/CodeBlock.test.js` — add tests for new/changed methods
- [ ] **3.2** Update `tests/CodeBlockConversion.test.js` — test toggle behavior, backtick trigger stripping

## Testing Strategy

| Scenario | Expected Behavior |
|----------|-------------------|
| Type `` ``` `` + Enter in paragraph | Creates empty code block (no backtick text inside) |
| Select language in code block | Syntax highlighting applies to correct block |
| Backspace in empty code block | Converts to paragraph |
| Click code button when in code block | Toggles back to paragraph |
| Click code button when in paragraph | Converts to code block |
| Bold/italic buttons when in code block | Disabled |
| Multiple code blocks, change language on 2nd | Only 2nd block re-highlights |

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Toggle logic affects other block types | Medium — other blocks (heading, list) may gain unexpected toggle behavior | Only toggle for block types that support it; test all block type conversions |
| Backspace conversion loses code content | High — user data loss if non-empty code block is converted | Only convert when `<code>` element is empty |
| `_element` reference becomes stale after DOM mutation | Medium — `refreshHighlighting` fails silently | Ensure `_element` is updated whenever the block DOM is replaced |
