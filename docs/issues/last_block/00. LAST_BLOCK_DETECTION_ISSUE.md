# New block appears in the wrong position after deleting all content and creating new blocks issue

This document describes an issue where a new block appears in the wrong position after deleting all content and creating new blocks in the editor.

## Scenario
- User opens the editor with prefilled blocks content.
- User presses `ctrl + a` to select all content.
- User presses `backspace` to delete all content.
- User types `# ` to create a heading block (Block 1). User presses `enter` to create a new block (Block 2).
- User starts typing regular text and hits `enter` to create a new block (Block 3).
- The new block (Block 3) appears between Block 1 and Block 2, instead of after Block 2.


The same goes with Header2, Header3, and so on.

## Root Cause
The focusin event handler in Editor.js:720 had a flawed fallback for when e.target.closest('.block') returns null. This happens because paragraph blocks (contenteditable="true") are nested inside the editor container (also contenteditable="true"), so the browser fires focusin with e.target = editor container, not the individual block.

The old code blindly defaulted to this.instance.querySelector('.block') — the first block in the DOM — which was the heading. This caused currentBlock to revert back to the heading every time focus moved to a paragraph after a heading conversion. When the user later pressed Enter, addDefaultBlock() inserted the new block after the heading (Block 1) instead of after the paragraph (Block 2).

## The Fix
In the focusin handler's else branch (Editor.js:724), instead of blindly picking the first block, the code now uses the current selection (window.getSelection().anchorNode) to find which block actually contains the cursor. It only falls back to the first block as a last resort.

## Why This Was an Architecture Flaw
The paragraph blocks and the editor container share the same contenteditable="true" attribute, creating an ambiguous focus boundary. The heading blocks don't have this problem because their outer div uses contenteditable="false" with an inner contenteditable="true" element. The focusin handler was written assuming e.target.closest('.block') would always find the correct block, but this assumption breaks for paragraph blocks nested in the contenteditable editor container.
