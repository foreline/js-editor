# Architecture Analysis & Improvement Proposals

## Status
Draft

## Summary
A comprehensive review of the BlockEditor's block-based architecture, covering all core components, their interactions, identified issues, and concrete improvement proposals. This analysis compares the current implementation against established best practices for contenteditable editors and proposes targeted fixes for the bugs blocking release—primarily around cursor positioning, block transformations, markdown conversions, and event timing.

## 2. Block-Based Architecture Analysis

### Block Contract (BaseBlock)

Every block class implements:

| Method/Property | Type | Purpose |
|----------------|------|---------|
| `handleKeyPress(evt, text)` | Instance | Custom key behavior (Tab in code, Ctrl+Space in task) |
| `handleEnterKey(evt)` | Instance | Enter key logic (new list item, exit list, etc.) |
| `toMarkdown()` | Instance | Serialize block to markdown |
| `toHtml()` | Instance | Serialize block to HTML |
| `applyTransformation()` | Instance | Convert current DOM block to this type |
| `renderToElement()` | Instance | Create fresh DOM element from block data |
| `getMarkdownTriggers()` | Static | Text patterns that auto-convert paragraphs |
| `matchesMarkdownTrigger(text)` | Static | Check if given text matches any trigger |
| `getToolbarConfig()` | Static | Button metadata (class, label, group, icon) |
| `getDisabledButtons()` | Static | Buttons to disable when this block is active |
| `canParseHtml(str)` | Static | Check if HTML string can be parsed as this block |
| `parseFromHtml(str)` | Static | Create block instance from HTML |
| `canParseMarkdown(str)` | Static | Check if markdown matches this block pattern |
| `parseFromMarkdown(str)` | Static | Create block instance from markdown |

### Block Hierarchy

```
BaseBlock
├── ParagraphBlock
├── HeadingBlock (abstract)
│   ├── H1Block
│   ├── H2Block
│   ├── H3Block
│   ├── H4Block
│   ├── H5Block
│   └── H6Block
├── ListBlock (abstract)
│   ├── UnorderedListBlock
│   ├── OrderedListBlock
│   └── TaskListBlock
├── CodeBlock
├── QuoteBlock
├── DelimiterBlock
├── TableBlock
└── ImageBlock
```

### Strengths

1. **Clean extensibility** — Adding a new block type requires only creating a class extending `BaseBlock`, implementing the contract, and registering in `BlockFactory`. No Editor/Toolbar changes needed.
2. **Self-describing blocks** — Each block defines its own triggers, toolbar config, and disabled buttons. The system is data-driven.
3. **Separation of rendering from state** — Blocks produce DOM via `renderToElement()` and can serialize to markdown/HTML independently.

### Weaknesses

1. **`applyTransformation()` is inconsistent across blocks** — HeadingBlock, CodeBlock, and ListBlock do direct DOM manipulation using `Editor.currentBlock` (static property). QuoteBlock delegates to `Toolbar.quote()` creating a circular dependency. There is no uniform interface for how transformations access the target block element.
2. **Static `Editor.currentBlock` usage in blocks** — Blocks should not reach into `Editor` statics. This couples blocks to a specific Editor implementation and breaks multi-editor scenarios.
3. **`_content` staleness** — Block objects created during `init()` have `_content` set at construction time. `toMarkdown()` and `toHtml()` read from `_content` which never gets updated as the user types. Content lives only in the DOM after initialization.
4. **`_nested` property unused** — Reserved for future use but adds unnecessary complexity to the constructor chain.

## 3. Core Component Analysis

### 3.1 Editor.js

**Role:** Lifecycle orchestrator, event wiring, block management, paste handling, serialization.

#### State Flags & Race Conditions

The editor relies on two boolean flags to prevent race conditions during block creation and conversion:

```
isCreatingBlock  — set in addDefaultBlock(), cleared via requestAnimationFrame
isConvertingBlock — set in convertBlockType(), cleared after conversion completes
```

**Problem:** These flags are editor-wide, not per-operation. If two rapid operations overlap (e.g., user types `# ` and immediately hits Enter), only one flag can be active. The `requestAnimationFrame` clearing is timing-dependent and may fire before or after the next input event depending on browser scheduling.

**Best Practice Alternative:** A state machine with explicit transitions would be more robust:
```
IDLE → CONVERTING → IDLE
IDLE → CREATING → IDLE
```
With a queue for operations that arrive during non-IDLE states.

#### Update Cycle

```
update() → 50ms debounce → _performUpdate()
  ├── updateBlockTimestamps()
  ├── ensureDefaultBlock() (guarded by flags)
  ├── emit EDITOR_UPDATED
  └── emit CONTENT_CHANGED (500ms debounce)
```

**Problem:** `update()` is called from many places: input handler, paste handler, KeyHandler, Toolbar.after(), convertBlockType(), and addDefaultBlock(). The 50ms debounce coalesces these into a single `_performUpdate()`, but this means the `ensureDefaultBlock()` guard only runs once per batch. If a conversion sets `isConvertingBlock = true` and the timeout fires before the flag is cleared, the guard is skipped. If it fires after, the guard runs with stale state.

**Observation:** The `ensureDefaultBlock()` mechanism has been fixed multiple times (see ISSUES.md — four separate entries). This suggests the root cause is the flag-based timing approach itself.

#### Serialization (getMarkdown / getHtml)

**Critical Bug (open in ISSUES.md):** `getMarkdown()` and `getHtml()` iterate `this.blocks[]` and call `block.toMarkdown()` / `block.toHtml()`. But `this.blocks` is only populated during `init()`. After initialization, the DOM is the source of truth, but `toMarkdown()` reads from stale `_content`. The user sees content in the editor but export returns empty or outdated strings.

**Root Cause:** There is no DOM-to-block synchronization. The architecture assumes blocks are stateful objects that track their own content, but in practice content lives in the DOM and blocks are created once then forgotten.

### 3.2 KeyHandler.js

**Role:** Static utility that dispatches keyboard events to appropriate block handlers.

**Pattern Issues:**
1. **Mixes static and instance patterns** — Receives `editorInstance` as parameter but accesses `Editor.currentBlock` (static property) directly. In a multi-editor scenario, `Editor.currentBlock` could refer to a different editor's block.
2. **Keybuffer grows unbounded** — Keys are pushed to `editorInstance.keybuffer` but only cleared explicitly via `clearKeyBuffer()`. No automatic trimming.
3. **Code block trigger detection** — Counts backticks in keybuffer by walking backward. This is fragile: if the user types backticks in normal text and then presses Enter, it could falsely detect a code block trigger.

**Best Practice:** KeyHandler should be instance-scoped (not static) and receive the editor instance at construction, storing it as a reference. Key detection should use the DOM content directly rather than maintaining a separate keybuffer.

### 3.3 Toolbar.js & ToolbarHandlers.js

**Role:** UI creation, action dispatch, button state management.

**Pattern Issues:**
1. **Singleton object, not a class** — `Toolbar` is a plain object with methods. This prevents instantiation per editor and makes multi-editor support difficult.
2. **`document.execCommand()` for inline formatting** — This API is deprecated. While it still works in browsers, it's not reliable across all contenteditable implementations and produces inconsistent markup.
3. **No button state tracking** — Bold/italic/underline active states are not reflected in the toolbar UI. The user cannot tell if the cursor is in bold text.
4. **`Toolbar.after()`** — Simply calls `editorInstance.update()`. This tight coupling means every toolbar action triggers a full update cycle whether or not content actually changed.

**Best Practice:** Toolbar should be a class instantiated per editor. Inline formatting should use the Selection API and Range manipulation with `document.createElement` / `surroundContents()` instead of `execCommand`. Active button states should be updated via `queryCommandState()` or by inspecting the DOM around the cursor.

### 3.4 Parser.js

**Role:** Static utility for HTML↔Markdown conversion and block extraction.

**Implementation:**
- Uses Showdown.js for markdown↔HTML conversion
- Custom preprocessing for task lists (transient `<task-item>` tags)
- Custom postprocessing for code blocks, blockquotes, headers
- Block extraction via regex-based HTML tag parsing

**Issues:**
1. **Heavy regex-based HTML manipulation** — Fragile for nested or malformed structures. The `extractHtmlBlocks()` tracks nesting depth but doesn't handle all edge cases (e.g., self-closing tags inside block-level elements).
2. **Task list preprocessing complexity** — Disabling Showdown's `tasklists` and implementing custom preprocessing adds significant code that could drift out of sync with markdown spec.
3. **No round-trip guarantee** — `markdown → HTML → blocks → HTML → markdown` does not always produce the original markdown. This manifests as content drift after paste operations.
4. **`>` character encoding** — The browser encodes `>` as `&gt;` in contenteditable. The quote block trigger `> ` doesn't match `&gt; ` (open bug in ISSUES.md).

### 3.5 Event System (EditorEventEmitter)

**Role:** Per-instance pub/sub with debouncing and throttling.

**Strengths:**
- Clean per-instance isolation
- Supports debounce, throttle, priority, once
- Structured event data with timestamps and sources

**Issues:**
1. **Debounce/throttle specified at emission site, not subscription** — The caller of `emit()` controls the timing, not the subscriber. This means different emission sites can debounce the same event with different intervals, causing inconsistent behavior.
2. **No event cancellation** — Subscribers cannot cancel or modify events. If a handler detects an invalid state, it cannot prevent other handlers from running.
3. **`Event.js` is a dead class** — The `Event` class in `src/Event.js` is empty. The real implementation is in `src/utils/eventEmitter.js`. This is confusing and should be cleaned up.

### 3.6 DebugTooltip.js

**Role:** Debug overlay showing block metadata.

**Issue:** The update loop (`startUpdateLoop()`) has been disabled with an early `return`. The tooltip only updates when explicitly called from `setCurrentBlock()` and `_performUpdate()`. This means the tooltip doesn't refresh if block content changes without focus changes.

## 4. Identified Root Causes of Release-Blocking Bugs

### 4.1 Cursor Position Issues

**Root Cause:** `applyTransformation()` implementations handle focus inconsistently:
- HeadingBlock creates an inner `<h1>` element and focuses it correctly
- ListBlock creates `<ul><li>` and focuses the first `<li>`
- CodeBlock creates `<pre><code>` but focus handling varies
- After conversion, `setCurrentBlock()` may fire before the new DOM structure exists

**Contributing Factor:** The `focusElement()` method uses `Range.selectNodeContents()` + `Range.collapse(false)` which works for text nodes but can fail for empty elements or elements with only child elements (the range collapses to the wrong position).

### 4.2 Block Transformation Bugs

**Root Causes:**
1. **Race condition between conversion and empty-editor guard** — `convertBlockType()` temporarily empties the block during DOM manipulation. The `input` event fires, sees an empty block, and `ensureDefaultBlock()` resets it before conversion completes. The `isConvertingBlock` flag was added to mitigate this but has timing gaps.
2. **QuoteBlock's circular `applyTransformation()`** — Calls `Toolbar.quote()` which may call back into the editor, creating unpredictable behavior.
3. **`>` encoding in contenteditable** — Quote block trigger won't work because the browser encodes `>` as `&gt;` before the trigger check runs.
4. **List triggers not matching** — Similar encoding/normalization issues may affect list triggers. The input handler normalizes `&nbsp;` but may not handle all browser-specific whitespace encoding.

### 4.3 Markdown Conversion Issues

**Root Causes:**
1. **Stale `_content` in block objects** — `toMarkdown()` reads from construction-time `_content`, not current DOM content. Export returns empty/stale data.
2. **No DOM → Block sync** — The architecture lacks a mechanism to sync DOM state back to block objects before serialization.
3. **Parser round-trip loss** — Custom preprocessing (task lists, code blocks) can alter content during the markdown → HTML → blocks pipeline.

### 4.4 Toolbar Button State Issues

**Root Cause:** `updateToolbarButtonStates()` only manages disabled/enabled state based on block type. It does not track inline formatting state (bold, italic, etc.). When a block is converted, the disabled button list from `getDisabledButtons()` is applied, but active toggle states for formatting buttons are never set.

### 4.5 Debouncing / Timing Issues

**Root Causes:**
1. **50ms update debounce** — Can coalesce critical operations. A conversion followed immediately by a focus change may produce only one update call, losing intermediate state.
2. **`requestAnimationFrame` for flag clearing** — RAF timing is frame-dependent (typically 16ms at 60fps) and not guaranteed to fire after all event handlers complete.
3. **Multiple callers of `update()`** — Input events, paste, KeyHandler, Toolbar.after() all trigger `update()`. The debounce helps but doesn't eliminate redundant work.

## 5. Comparison with Best Practices

### 5.1 How a Modern Block Editor Should Be Designed

The following comparison draws from established patterns in editors like ProseMirror, Slate.js, and Editor.js:

| Aspect | Current Architecture | Best Practice |
|--------|---------------------|---------------|
| **State ownership** | DOM is source of truth; block objects are stale after init | Separate state model; DOM is a view. Changes update the model first, then re-render |
| **Transformations** | Directly mutate DOM in `applyTransformation()` | Produce a new state via commands; framework applies diff to DOM |
| **Cursor management** | Manual Selection/Range API after each DOM change | State includes cursor position; rendering restores cursor from state |
| **Undo/Redo** | `document.execCommand('undo')` (browser-native) | Custom undo stack recording state snapshots or operations |
| **Race conditions** | Boolean flags (`isCreatingBlock`, `isConvertingBlock`) | Transaction-based operations: changes batched and applied atomically |
| **Inline formatting** | `document.execCommand('bold')` | Marks/decorations in state model; rendered as `<strong>` |
| **Toolbar state** | Manually query block type for disabled buttons | Derive toolbar state from current selection position in state model |
| **Serialization** | Walk DOM blocks, each calls `toMarkdown()` with stale data | Serialize from state model (always current) |
| **Multi-editor** | Static singletons (Toolbar, KeyHandler) with instance shims | All components instantiated per editor; no global state |
| **Event timing** | Debounce at both emission and update level | Single change event from state mutations; no timing issues |

### 5.2 Critical Gaps

#### Gap 1: No State Model (DOM as Source of Truth)

This is the root cause of many issues. Without a state model:
- Serialization requires DOM traversal and content extraction each time
- Block objects go stale immediately after construction
- Race conditions arise because DOM mutations trigger browser events that interfere with in-progress operations
- Undo/redo relies on browser's contenteditable undo stack which doesn't understand block boundaries

**Impact:**  Affects export bugs, conversion races, cursor restoration

**Recommendation:** Introduce a lightweight state model where each block has a `content` property that is synced from the DOM on every input event. This is less invasive than a full virtual DOM but fixes the serialization and staleness issues.

#### Gap 2: Static Singletons vs Instance Scoping

`KeyHandler`, `Toolbar`, `Parser` are static/singleton. This means:
- Multi-editor scenarios share toolbar state
- `Editor.currentBlock` is a static property (only one can be "current" globally)
- KeyHandler uses `Editor.keybuffer` statically

**Impact:** Broken multi-editor support, confusing mixing of static and instance patterns

**Recommendation:** Convert KeyHandler and Toolbar to instance-scoped classes. Each `Editor` instance should own its `KeyHandler` and `Toolbar`. `Parser` can remain static since it's stateless.

#### Gap 3: No Transaction/Command Pattern

Currently, a block conversion involves:
1. Setting a flag (`isConvertingBlock = true`)
2. Creating a new block via `BlockFactory`
3. Calling `block.applyTransformation()` which mutates the DOM
4. Clearing trigger text from the DOM
5. Setting focus
6. Clearing the flag
7. Calling `update()`

If any step fails or an event fires between steps, the editor can end up in an inconsistent state.

**Recommendation:** Wrap multi-step operations in a transaction:
```javascript
editor.transaction(() => {
    // All DOM mutations batched
    // Input/update events suppressed
    // Executes atomically
    // On error: rollback
});
```

#### Gap 4: `document.execCommand` for Formatting

`document.execCommand` is deprecated and produces inconsistent HTML across browsers (some use `<b>`, others `<strong>`, some wrap in `<span style="font-weight:bold">`).

**Recommendation:** Replace with Selection/Range-based formatting that produces consistent semantic markup.

#### Gap 5: `applyTransformation()` Inconsistency

Each block implements `applyTransformation()` differently:
- HeadingBlock: Direct DOM manipulation, accesses `Editor.currentBlock` (static)
- CodeBlock: Direct DOM manipulation, accesses `Editor.currentBlock` (static)
- ListBlock: Direct DOM manipulation, accesses `Editor.currentBlock` (static), also gets editor instance
- QuoteBlock: Delegates to `Toolbar.quote()` (circular dependency!)
- DelimiterBlock: Direct DOM manipulation
- Some blocks return values, others don't

**Recommendation:** Standardize the interface:
```javascript
applyTransformation(targetElement, editorInstance) {
    // Receives the block element and editor instance as parameters
    // No static Editor access
    // No Toolbar calls
    // Returns the new editable element (for focus management)
}
```

## 6. Specific Bug Fix Recommendations

### 6.1 Export Functions Return Empty Strings

**Bug:** `editor.getMarkdown()` / `editor.getHtml()` return empty/stale content.

**Fix:** Replace stale block iteration with DOM-based content extraction:
```javascript
getMarkdown() {
    const blocks = this.instance.querySelectorAll('.block');
    const markdownParts = [];
    for (const blockEl of blocks) {
        const blockType = blockEl.getAttribute('data-block-type');
        const BlockClass = BlockFactory.getBlockClass(blockType);
        // Extract current content from DOM, not from stored _content
        const currentContent = this.extractBlockContent(blockEl);
        const tempBlock = new BlockClass(currentContent, blockEl.innerHTML);
        markdownParts.push(tempBlock.toMarkdown());
    }
    return markdownParts.join('\n\n');
}
```

### 6.2 Code Block Markdown Trigger Not Working

**Bug:** Typing ``` + Enter doesn't create a code block.

**Fix:** `CodeBlock.applyTransformation()` is a no-op. Implement it following the HeadingBlock pattern (direct DOM manipulation, no Toolbar call).

### 6.3 Quote Block Markdown Trigger Not Working

**Bug:** `> ` doesn't trigger quote block because browser encodes `>` as `&gt;`.

**Fix:** Add `&gt;` to the normalization step in the input handler's trigger regex, or decode HTML entities before trigger matching:
```javascript
const textContent = normalized
    .replace(/&gt;/g, '>')
    .replace(/&lt;/g, '<')
    .replace(/^\s+/, '');
```

### 6.4 QuoteBlock Circular Dependency

**Bug:** `QuoteBlock.applyTransformation()` calls `Toolbar.quote()`.

**Fix:** Implement direct DOM transformation matching the HeadingBlock pattern:
```javascript
applyTransformation() {
    const currentBlock = this.constructor._getCurrentBlock();
    if (!currentBlock) return;
    currentBlock.setAttribute('data-block-type', 'quote');
    currentBlock.className = 'block block-quote';
    const blockquote = document.createElement('blockquote');
    blockquote.setAttribute('contenteditable', 'true');
    blockquote.textContent = currentBlock.textContent;
    currentBlock.innerHTML = '';
    currentBlock.appendChild(blockquote);
    blockquote.focus();
}
```

### 6.5 List Markdown Triggers Not Working

**Bug:** `- `, `* `, `1. ` don't convert paragraph to list.

**Fix:** Debug the input handler's trigger regex and normalization. Ensure that the regex matches after HTML entity decoding and that `checkAndConvertBlock()` correctly detects list triggers. The `requestAnimationFrame` wrapper may fire too late — test with synchronous execution.

### 6.6 Unordered List Rendering in Empty Editor

**Bug:** Creating a UL in empty editor – list appears then disappears.

**Root Cause:** The conversion temporarily empties the block, triggering `ensureDefaultBlock()` which resets the editor. The `isConvertingBlock` flag should prevent this, but timing gaps exist.

**Fix:** Ensure `isConvertingBlock` is set before any DOM mutation begins and cleared only after the conversion is fully complete including the `requestAnimationFrame` focus callback.

### 6.7 Block Conversion Performance

**Bug:** Converting paragraph to UL causes multiple `update()` / `html2md()` calls.

**Fix:** The `update()` debounce should prevent this, but multiple callers may reset the timeout. Add a guard:
```javascript
update() {
    if (this._updateScheduled) return;
    this._updateScheduled = true;
    if (this._updateTimeout) clearTimeout(this._updateTimeout);
    this._updateTimeout = setTimeout(() => {
        this._updateScheduled = false;
        this._performUpdate();
    }, 50);
}
```

## 7. Prioritized Implementation Plan

### Phase 1 — Critical Bug Fixes (Immediate)

These fixes address open bugs in ISSUES.md and can be done without architectural changes:

- [ ] **Add toolbar active state tracking** — Use `document.queryCommandState()` or DOM inspection to reflect bold/italic state

- [ ] **Replace `document.execCommand`** — Use Selection/Range API for inline formatting

### Phase 4 — Advanced (Long-term)

- [ ] **Introduce lightweight state model** — Block state objects synced from DOM; serialize from state not DOM

- [ ] **Parser robustness** — Replace regex-based HTML extraction with DOM-based parsing

## 10. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Standardizing `applyTransformation()` breaks existing block implementations | High — all blocks need updating | Do it incrementally; keep old signature as fallback during migration |
| DOM-to-block sync introduces performance overhead | Medium — sync on every input event | Sync lazily (only before serialization) or use MutationObserver with batching |
| Removing `document.execCommand` breaks inline formatting | High — bold/italic/etc. stop working | Implement and test new formatting before removing old approach |
| Converting Toolbar/KeyHandler to classes breaks backward compatibility | Medium — consumers of static API break | Maintain static shims that delegate to the first instance (already done for Editor) |
| Transaction wrapper adds complexity | Low — scoped change | Keep implementation minimal: just flag-based event suppression |
| Multi-editor support changes expose hidden static dependencies | Medium — bugs in edge cases | Add multi-editor E2E tests before migration |
