## 5. Additional Observations

### 5.1 `applyTransformation()` Circular Dependencies — Two Remaining

The original analysis identified circular dependencies in block `applyTransformation()` methods calling `Toolbar` methods. Most were fixed, but two remain:

- **`DelimiterBlock.applyTransformation()`** → ✅ Fixed — now performs direct DOM transformation
- **`ParagraphBlock.applyTransformation()`** → ✅ Fixed — now performs direct DOM transformation

All `applyTransformation()` circular dependencies have been resolved.

### 5.2 DebugTooltip Update Loop — Enabled via MutationObserver

`DebugTooltip.startUpdateLoop()` now uses a `MutationObserver` (watching `childList`, `attributes`, `characterData` on the editor container subtree) plus a `focusin` event listener to reactively update the tooltip. Updates are coalesced via `requestAnimationFrame` to avoid redundant DOM reads during rapid mutations. Arrow-key navigation between blocks triggers immediate tooltip refresh via the `focusin` bubble.

### 5.3 Multi-Editor Support — Still Fragile

While the static-to-instance migration for `Editor` is largely done (static methods are deprecated shims), `Toolbar` has been converted to a per-editor class instance (stored as `editor.toolbar`), and `KeyHandler` has been converted to a per-editor class instance (stored as `editor.keyHandler`). Multi-editor support is now structurally sound for toolbar and key handling.

### 5.4 `document.execCommand` — Increasing Browser Risk

The deprecated `execCommand` API is used in 4 locations:
1. `Toolbar.js` — inline formatting (bold, italic, underline, strikethrough)
2. `Toolbar.js` — undo/redo  
3. `CodeBlock.js` — tab insertion (`insertText`)
4. `ToolbarHandlers.js` — `defaultParagraphSeparator`

While browsers still support it, no guarantees exist for future versions. Replacement with Selection/Range API is recommended.
