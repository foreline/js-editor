# Inline Markdown Replacements — Analysis & Proposal

## Status
Draft

## Summary

This document analyzes why inline markdown patterns (`**bold**`, `*italic*`, `__bold__`, `_italic_`, `` `code` ``) do not trigger live transformations as users type, and proposes an architecture to implement them. The current markdown replacement system only supports **block-level triggers** (e.g., `# ` → heading, `- ` → list). Inline formatting is exclusively available via toolbar buttons using `document.execCommand()`. This is a fundamentally different class of transformation that requires a new subsystem.

## Problem Analysis

### Why Current Patterns Don't Work

The existing markdown trigger system is designed exclusively for **block-level conversions** — transforming a paragraph block into a different block type. Here's why it cannot handle inline patterns:

#### 1. Block-Level Architecture Only

The trigger pipeline works as follows:

```
User types in a paragraph block
  → `input` event fires in Editor.js (line 622)
  → Text is checked against a regex pre-filter (line 670):
      /^(#{1,6}\s|[\*\-\+]\s|[\*\-]\s*\[[x\s]\]\s*|\d+[\.)]?\s|>\s|```|~~~)/
  → BlockFactory.findBlockClassForTrigger(text) searches all block classes
  → If match found: convertBlockType() replaces the block entirely
```

Key constraints:
- **Only paragraphs are checked** — the system explicitly skips non-paragraph blocks (`blockType === 'p' || blockType === 'paragraph'`).
- **Triggers must be at the start of the text** — `text.startsWith(trigger)` is used for matching.
- **The entire block is converted** — the operation replaces the block's type and strips the trigger prefix. There is no concept of formatting a *portion* of text within a block.
- **The pre-filter regex does not match inline patterns** — patterns like `**text**` anywhere in the content are never even evaluated.

#### 2. No Inline Pattern Detection Infrastructure

The editor has no mechanism to:
- Detect **closing** markers (e.g., the second `**` in `**bold**`).
- Find **matching pairs** of markers around text content.
- Apply formatting to a **text range** within a block (wrap in `<strong>`, `<em>`, `<code>`).
- Remove marker characters while preserving and formatting the enclosed text.

#### 3. Inline Formatting is Toolbar-Only

Bold, italic, underline, and strikethrough are applied exclusively through `document.execCommand()` in `Toolbar.js`:

```javascript
bold()  { document.execCommand('bold'); }
italic() { document.execCommand('italic'); }
```

These operate on the current selection, not on pattern detection in typed text.

### Fundamental Difference: Block vs. Inline Triggers

| Aspect | Block-Level Triggers | Inline Markdown Patterns |
|--------|---------------------|-------------------------|
| **Position** | Start of block only | Anywhere in text |
| **Scope** | Entire block is replaced | Portion of text is formatted |
| **Detection** | Prefix match (`startsWith`) | Paired markers around content |
| **Timing** | On any input event | Only when closing marker is typed |
| **DOM change** | Block element type changes | Text nodes get wrapped in formatting tags |
| **Examples** | `# `, `- `, `` ``` `` | `**bold**`, `*italic*`, `` `code` `` |

## Proposed Solution

### Overview

Introduce an **InlineMarkdownHandler** — a new module that monitors text input within blocks and detects completed inline markdown patterns, replacing them with formatted HTML in real time.

### Supported Patterns

| Pattern | Result | HTML Output | Priority |
|---------|--------|-------------|----------|
| `**text**` | **bold** | `<strong>text</strong>` | High |
| `__text__` | **bold** | `<strong>text</strong>` | High |
| `*text*` | *italic* | `<em>text</em>` | High |
| `_text_` | *italic* | `<em>text</em>` | High |
| `` `text` `` | `code` | `<code>text</code>` | High |
| `~~text~~` | ~~strikethrough~~ | `<del>text</del>` | Medium |

### Architecture

#### New Component: `InlineMarkdownHandler`

**File**: `src/InlineMarkdownHandler.js`

This handler is instantiated by each `Editor` instance and hooks into the `input` event pipeline, running **after** block-level trigger checks.

```
input event
  → Block-level trigger check (existing - checkAndConvertBlock)
  → Inline markdown pattern check (NEW - InlineMarkdownHandler)
```

#### Pattern Detection Strategy

Two viable approaches were considered:

##### Option A: Input Event + Text Scanning (Recommended)

On every `input` event, scan the current text node (or the block's text content) for completed inline patterns using regex:

```javascript
const INLINE_PATTERNS = [
    { pattern: /\*\*(.+?)\*\*/, tag: 'strong' },           // **bold**
    { pattern: /__(.+?)__/, tag: 'strong' },                 // __bold__
    { pattern: /(?<!\*)\*(?!\*)(.+?)(?<!\*)\*(?!\*)/, tag: 'em' },  // *italic* (not **)
    { pattern: /(?<!_)_(?!_)(.+?)(?<!_)_(?!_)/, tag: 'em' }, // _italic_ (not __)
    { pattern: /`([^`]+)`/, tag: 'code' },                   // `code`
    { pattern: /~~(.+?)~~/, tag: 'del' },                    // ~~strikethrough~~
];
```

**Pros**:
- Simpler implementation — works on text content directly.
- No need to track keystrokes or maintain state between events.
- Handles paste and other non-keyboard input sources.
- Naturally handles the priority order (check `**` before `*`).

**Cons**:
- Slightly more expensive per input event (regex evaluation).
- Must carefully handle cursor repositioning after DOM manipulation.
- Must avoid re-triggering on already-formatted content.

##### Option B: Keystroke-Based State Machine

Track opening markers in a buffer. When a closing marker matches, apply formatting.

**Pros**:
- Lower per-event cost (only checks when specific characters are typed).
- More explicit control over detection logic.

**Cons**:
- More complex state management.
- Doesn't work with paste or drag-and-drop input.
- Must handle edge cases like abandoned markers (user types `**` but never closes).
- State gets complicated with nested/overlapping patterns.

#### Recommended Approach: Option A (Text Scanning)

The input event + text scanning approach is recommended because:
1. It's consistent with how block-level triggers already work (text is scanned on input).
2. It handles all input methods (keyboard, paste, speech-to-text, etc.).
3. It avoids complex state management.
4. The performance cost of regex is negligible for single-block text content.

### Detailed Design

#### 1. InlineMarkdownHandler Class

```javascript
// src/InlineMarkdownHandler.js

export class InlineMarkdownHandler {
    constructor(editorInstance) {
        this.editorInstance = editorInstance;
    }

    // Pattern definitions, checked in order (longer markers first)
    static PATTERNS = [
        { regex: /\*\*(.+?)\*\*/, tag: 'strong', marker: '**' },
        { regex: /__(.+?)__/, tag: 'strong', marker: '__' },
        { regex: /(?<!\*)\*(?!\*)(.+?)(?<!\*)\*(?!\*)/, tag: 'em', marker: '*' },
        { regex: /(?<!_)_(?!_)(.+?)(?<!_)_(?!_)/, tag: 'em', marker: '_' },
        { regex: /`([^`]+)`/, tag: 'code', marker: '`' },
        { regex: /~~(.+?)~~/, tag: 'del', marker: '~~' },
    ];

    /**
     * Check text content of a block for inline markdown patterns
     * and apply formatting if found.
     * @param {HTMLElement} blockElement - The block DOM element
     * @returns {boolean} - Whether a replacement was made
     */
    checkAndApply(blockElement) {
        // Skip code blocks — they should not get inline formatting
        if (blockElement.dataset.blockType === 'code') return false;

        // Walk text nodes within the editable element
        const editableEl = this._getEditableElement(blockElement);
        if (!editableEl) return false;

        // Get the text content and check for patterns
        // Use textContent to avoid matching already-wrapped HTML
        const textNodes = this._getTextNodes(editableEl);

        for (const textNode of textNodes) {
            const text = textNode.nodeValue;
            for (const { regex, tag } of InlineMarkdownHandler.PATTERNS) {
                const match = regex.exec(text);
                if (match && match[1]) {
                    this._applyFormat(textNode, match, tag);
                    return true;
                }
            }
        }

        return false;
    }

    _getEditableElement(blockElement) {
        // Find the contenteditable element inside the block
        return blockElement.querySelector('[contenteditable]') || blockElement;
    }

    _getTextNodes(element) {
        const nodes = [];
        const walker = document.createTreeWalker(
            element,
            NodeFilter.SHOW_TEXT,
            null,
            false
        );
        while (walker.nextNode()) {
            nodes.push(walker.currentNode);
        }
        return nodes;
    }

    _applyFormat(textNode, match, tag) {
        const before = textNode.nodeValue.substring(0, match.index);
        const content = match[1];
        const after = textNode.nodeValue.substring(match.index + match[0].length);

        // Create the formatted element
        const formatted = document.createElement(tag);
        formatted.textContent = content;

        // Build replacement fragment
        const parent = textNode.parentNode;

        if (before) {
            parent.insertBefore(document.createTextNode(before), textNode);
        }

        parent.insertBefore(formatted, textNode);

        if (after) {
            parent.insertBefore(document.createTextNode(after), textNode);
        }

        parent.removeChild(textNode);

        // Place cursor after the formatted element
        this._placeCursorAfter(formatted);
    }

    _placeCursorAfter(element) {
        const selection = window.getSelection();
        const range = document.createRange();

        // If there's a text node after the element, position cursor at its start
        if (element.nextSibling && element.nextSibling.nodeType === Node.TEXT_NODE) {
            range.setStart(element.nextSibling, 0);
        } else {
            // Create an empty text node to place cursor after formatted element
            const emptyText = document.createTextNode('\u200B'); // zero-width space
            element.parentNode.insertBefore(emptyText, element.nextSibling);
            range.setStart(emptyText, 1);
        }

        range.collapse(true);
        selection.removeAllRanges();
        selection.addRange(range);
    }
}
```

#### 2. Integration with Editor.js

The handler hooks into the existing `input` event listener, running after block-level trigger checks:

```javascript
// In Editor.js constructor or init
this._inlineMarkdownHandler = new InlineMarkdownHandler(this);

// In the input event listener, after block-level trigger check:
if (block) {
    const blockType = block.getAttribute('data-block-type');
    if (blockType === 'p' || blockType === 'paragraph') {
        // ... existing block-level trigger check ...
    }

    // NEW: Inline markdown pattern check (for all non-code block types)
    if (blockType !== 'code') {
        requestAnimationFrame(() => {
            if (this._inlineMarkdownHandler.checkAndApply(block)) {
                this.update();
            }
        });
    }
}
```

#### 3. Edge Cases to Handle

| Edge Case | Handling |
|-----------|----------|
| Already-formatted text (e.g., inside `<strong>`) | Only scan raw text nodes, skip nodes inside formatting tags |
| Incomplete patterns (e.g., user typed `**` but no closing `**` yet) | No action — patterns only match when both markers are present |
| Code blocks | Skip entirely — no inline formatting inside code |
| Escaped markers (e.g., `\*not italic\*`) | Consider supporting backslash escaping in a future iteration |
| Nested formatting (`***bold italic***`) | Handle in a future iteration; initial version handles one level |
| Marker-like content in URLs or filenames | The lazy quantifier `.+?` limits false positives; further refinement possible |
| `_` inside words (e.g., `snake_case_name`) | Use word-boundary aware patterns or `literalMidWordUnderscores` behavior (skip `_` surrounded by word chars) |
| Large text content | Regex is evaluated per text node, not per block — performance impact is minimal |

#### 4. Mid-Word Underscore Protection

Following the Showdown `literalMidWordUnderscores: true` convention already used in Parser.js, underscores in the middle of words should NOT trigger italic. The `_italic_` pattern regex should require whitespace or start/end-of-string around the outer underscores:

```javascript
// Updated pattern: _ must be at word boundary
{ regex: /(?:^|(?<=\s))_(?!_)(.+?)(?<!_)_(?:$|(?=\s))/, tag: 'em', marker: '_' }
```

### Interaction with Existing Features

- **Toolbar formatting**: Works independently. Users can still select text and click Bold/Italic. The inline markdown handler adds an *additional* way to achieve the same result.
- **Block conversion**: No conflict. Block-level triggers (`# `, `- `) are checked first. Inline patterns only run on text that didn't trigger a block conversion.
- **Markdown export** (`toMarkdown()`): Should work naturally — `<strong>` and `<em>` tags are already handled by the HTML→Markdown pipeline.
- **Paste handling**: The inline handler runs on `input` events, which fire after paste. Pasted markdown patterns will be detected and converted.
- **Undo/redo**: `document.execCommand` provides built-in undo support. Since this approach uses direct DOM manipulation instead of `execCommand`, undo behavior needs consideration. Consider wrapping the replacement in a mutation that the browser's undo stack can track, or integrating with a future custom undo system.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Performance overhead on every input event | Slight delay in typing responsiveness | Pattern check is per-text-node, not per-block innerHTML; use early exit if no marker characters present |
| Cursor jumps after DOM manipulation | Poor UX, frustrating editing | Careful cursor repositioning using Selection/Range API; test thoroughly across browsers |
| Conflict with `*` trigger for unordered list | `* ` at block start could match both list trigger and start of `*italic*` | Block-level triggers run first and skip text; inline check only runs if no block conversion occurred |
| Undo/redo behavior | Users expect Ctrl+Z to revert formatting | Document as known limitation; address with custom undo system (separate concern) |
| False positives in URLs or technical content | Content like `http://example.com/path_to_file` triggers italic | Use word-boundary checks for `_`; consider context-aware filtering |
| `***` ambiguity (bold+italic vs triple marker) | Unexpected formatting | Handle `***text***` as bold+italic in future iteration; initial release handles `**` and `*` separately |
