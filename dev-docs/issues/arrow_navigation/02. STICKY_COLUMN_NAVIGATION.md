# Sticky Column Arrow Navigation

## Status
Implemented

## Summary
Improve ArrowUp/ArrowDown cross-block navigation to preserve the cursor's horizontal offset, following the behaviour of IDEs and desktop text editors. When navigating into an empty block, the offset is remembered so that subsequent arrow presses restore the cursor to the original column in the next non-empty block.

## Problem
After the initial fix in `01. ANALYSIS_AND_PROPOSAL.md`, arrow navigation between blocks works but behaves inconsistently:

- Pressing ArrowDown always places the cursor at the **start** of the next block, regardless of where it was in the current block.
- Pressing ArrowUp always places the cursor at the **end** of the previous block.
- When passing through an empty block, the position resets and subsequent navigation loses track of the original column.

This feels unnatural compared to IDEs (VS Code, IntelliJ) and rich text editors (Google Docs, Notion) where the cursor column is preserved across vertical moves.

## Expected Behaviour (IDE model)

| Scenario | Expected cursor placement |
|----------|--------------------------|
| Cursor at offset **N** in block A, ArrowDown to block B (non-empty) | Offset **min(N, len(B))** — same column, or end if B is shorter |
| Cursor at offset **N** in block A, ArrowDown to block B (empty) | Offset **0** (only valid position), but **N** is remembered |
| Previously passed through an empty block with remembered offset **N**, ArrowDown to block C (non-empty) | Offset **min(N, len(C))** — original column restored |
| Any non-arrow key resets the remembered offset | Offset memory cleared |

This is the "sticky column" or "desired column" pattern used universally in mono-spaced and rich-text editors.

## Design

### New state: `_desiredOffset` on KeyHandler
A single numeric property that tracks the target cursor offset across consecutive ArrowUp/ArrowDown presses. It is:
- **Set** to the current cursor offset when the first arrow navigation occurs (only if not already set).
- **Preserved** when landing on an empty block (the offset cannot be honoured, but it's kept for the next move).
- **Cleared** on any non-arrow keydown event (typing, Enter, Backspace, etc.) and on mouse clicks.

### New method: `placeCursorAtOffset(element, offset)` on Editor
Places the cursor at a specific character offset within an element, clamping to the element's text length if it's shorter.

### Changes to `handleArrowKey()`
Instead of unconditionally using `placeCursorAtEnd` (up) / `placeCursorAtStart` (down), the method now:
1. Reads the current cursor offset within the block.
2. On first arrow press, stores it as `_desiredOffset`.
3. Navigates to the sibling block, placing the cursor at `min(_desiredOffset, siblingTextLength)`.
4. If the sibling is empty, preserves `_desiredOffset` for subsequent presses.

### Changes to `handleSpecialKeys()` / `handleKeyPress()`
Non-arrow keys reset `_desiredOffset = null` to restart the sticky column on the next arrow sequence.

## Alternatives Considered
- **Per-block offset tracking**: Storing column per block — over-engineered, not how editors work.
- **Using pixel-based positioning**: Computing X coordinates via `getBoundingClientRect` — fragile with variable-width fonts and inline formatting; unnecessary for single-line blocks.
