# Empty Block Arrow Navigation — Analysis and Proposal

## Status
Draft

## Summary
Users cannot navigate into empty blocks using arrow keys. This document identifies the root cause, proposes a solution, describes alternatives considered, and provides rationale for the chosen approach.

## Problem Statement

When a user presses **Enter** at the end of a block, a new empty paragraph block is created below the cursor. The cursor is placed in that new block via `requestAnimationFrame` + `focusElement()`. However, if the user subsequently navigates away (e.g., clicks another block or arrows up), they **cannot arrow back into the empty block**. The cursor skips over it entirely.

### Reproduction Steps
1. Open the editor and type some text in a paragraph block.
2. Press **Enter** to create a new empty block below.
3. Arrow up to the previous block.
4. Arrow down — the cursor does **not** land in the empty block; it jumps past it.

### Expected Behavior
Arrow keys should allow the cursor to enter any block, including empty blocks, so users can start typing without resorting to mouse clicks.

---

## Root Cause Analysis

### 1. No Explicit Arrow Key Handling

The `KeyHandler` class (`src/KeyHandler.js`) only handles four key types:
- **Enter** → `handleEnterKey()`
- **Backspace** → `handleBackspaceKey()`
- **Delete** → `handleDeleteKey()`
- **Tab** → delegated to block types

**ArrowUp, ArrowDown, ArrowLeft, ArrowRight are not handled at all.** Navigation between blocks relies entirely on the browser's native `contenteditable` caret movement.

### 2. Browser's Native Caret Navigation Skips Empty Elements

The editor container (`div.editor`) has `contenteditable="true"`. Block divs inside it do **not** set `contenteditable` themselves (except special block types like headings and code blocks that use nested contenteditable). Blocks inherit editability from the parent.

When a block is empty (`innerHTML === ""`), the browser has no text node or `<br>` element to place the caret on. The native arrow key navigation algorithm looks for the next valid **caret position**, which requires either:
- A text node with characters, or
- A `<br>` element acting as a line break

An empty `<div></div>` inside a contenteditable container provides **no caret position**, so the browser's caret movement algorithm skips it entirely.

### 3. CSS `:empty` Placeholder is Visual Only

The CSS rule `.editor .block:empty::before` renders a placeholder via `content: attr(data-placeholder)`. This is a CSS pseudo-element — it is **not part of the DOM**, so it does not create a valid caret position for the browser.

### 4. The `min-height` Rule is Also Insufficient

The CSS `.editor .block { min-height: calc(1em * var(--editor-line-height)); }` ensures the empty block occupies vertical space visually, but does not solve the caret navigation issue since there is still no text node or `<br>` inside the element.

### 5. `focusin` Event Works Only After Focus

The editor listens for `focusin` events to track the active block (`setCurrentBlock()`). This works when the browser successfully focuses an element — but if the browser cannot place the caret in an empty block, no `focusin` event fires, so the tracking never activates.

---

## Proposed Solution: Add ArrowUp/ArrowDown Handlers in KeyHandler

### Approach

Add explicit `ArrowUp` and `ArrowDown` key handlers in `handleSpecialKeys()` within `KeyHandler.js`. When the cursor is at the first line of a block (ArrowUp) or the last line of a block (ArrowDown), manually move focus to the previous or next sibling block.

### Why This Approach

1. **Directly solves the problem** — when the browser can't navigate into an empty block, the explicit handler takes over and forces focus + cursor placement.
2. **Non-breaking** — only intercepts when the cursor is at a block boundary, otherwise lets the browser handle arrow navigation normally (for multi-line blocks, intra-block navigation works as before).
3. **Consistent with existing patterns** — mirrors how Enter, Backspace, and Delete already intercept at block boundaries.
4. **Handles both empty and non-empty blocks** — provides reliable cross-block navigation regardless of content.

### Design

#### KeyHandler.js Changes

Add to `handleSpecialKeys()`:
```javascript
if ('ArrowUp' === e.key || 'ArrowDown' === e.key) {
    return this.handleArrowKey(e);
}
```

New method `handleArrowKey(e)`:
```javascript
handleArrowKey(e) {
    const currentBlock = this.editorInstance.currentBlock;
    if (!currentBlock || !currentBlock.isConnected) return;

    const isUp = e.key === 'ArrowUp';
    const selection = window.getSelection();
    if (!selection.rangeCount) return;
    const range = selection.getRangeAt(0);

    // Only intervene when the cursor is at the boundary of the block
    const atBoundary = isUp
        ? KeyHandler.isCursorAtStartOfBlock(currentBlock, range)
        : KeyHandler.isCursorAtEndOfBlock(currentBlock, range);

    if (!atBoundary) return; // Let browser handle intra-block movement

    const sibling = isUp
        ? currentBlock.previousElementSibling
        : currentBlock.nextElementSibling;

    if (!sibling || !sibling.classList.contains('block')) return;

    e.preventDefault();

    this.editorInstance.setCurrentBlock(sibling);
    const editable = this.editorInstance.findEditableElementInBlock(sibling);

    if (isUp) {
        this.editorInstance.placeCursorAtEnd(editable || sibling);
    } else {
        this.editorInstance.placeCursorAtStart(editable || sibling);
    }
}
```

#### Editor.js Changes

Add `placeCursorAtStart()` method (counterpart to existing `placeCursorAtEnd()`):
```javascript
placeCursorAtStart(element) {
    if (!element) return;
    try {
        const selection = window.getSelection();
        const range = document.createRange();
        range.setStart(element, 0);
        range.collapse(true);
        selection.removeAllRanges();
        selection.addRange(range);
    } catch (error) {
        // Silently fail for cursor positioning
    }
}
```

### Block Contract Impact
- **No changes** to the block contract. No new instance or static methods required on blocks.

### DOM Changes
- **None.** No new attributes, classes, or structural modifications.

### API Surface
- One new instance method on Editor: `placeCursorAtStart(element)` — a natural counterpart to the existing `placeCursorAtEnd()`.
- One new method on KeyHandler: `handleArrowKey(e)` — internal, not part of the public API.

## Alternatives Considered

### Alternative A: Insert `<br>` into Empty Blocks

**Approach:** When a block is created empty, insert `<br>` as content so the browser has a valid caret position.

**Pros:**
- Simple implementation.
- Works with browser's native navigation.

**Cons:**
- Changes the DOM content of empty blocks — `<br>` would appear in `innerHTML`, affecting `toMarkdown()`, `toHtml()`, serialization, and the `:empty` CSS pseudo-selector.
- Requires careful stripping of the `<br>` during export and checking.
- Interacts poorly with `isBlockEmpty()` checks that currently treat `<br>`-only blocks as empty.
- Could cause unexpected behavior during paste and conversion flows.

**Verdict:** Rejected — too many side effects across the codebase.

### Alternative B: Insert Zero-Width Space (`\u200B`) into Empty Blocks

**Approach:** Place an invisible character in empty blocks so the browser recognizes a valid caret position.

**Pros:**
- Doesn't visually change anything.
- Provides a caret position for native navigation.

**Cons:**
- The zero-width space is real content — it appears in `textContent`, breaks `isBlockEmpty()` checks, and leaks into exported Markdown/HTML unless carefully stripped everywhere.
- Users may inadvertently copy invisible characters.
- Complex to maintain across all block creation and conversion paths.

**Verdict:** Rejected — fragile, adds hidden complexity throughout the system.

### Alternative C: Use Only CSS `min-height` + `user-modify`

**Approach:** Rely purely on CSS to make empty blocks focusable.

**Pros:**
- No JavaScript changes.

**Cons:**
- CSS cannot create caret positions inside empty elements in contenteditable.
- `user-modify` / `-webkit-user-modify` are non-standard and deprecated.
- Does not work across all browsers.

**Verdict:** Rejected — technically infeasible.

---

## Rationale for Chosen Approach

The explicit arrow key handler approach is the most **surgical** fix:

1. **Zero side effects on content** — no hidden characters, no DOM changes, no serialization impact.
2. **Follows existing patterns** — the `KeyHandler` already intercepts Enter/Backspace/Delete at block boundaries with the same cursor-position detection logic.
3. **Reuses existing infrastructure** — `isCursorAtStartOfBlock()`, `isCursorAtEndOfBlock()`, `findEditableElementInBlock()`, `setCurrentBlock()`, and `placeCursorAtEnd()` already exist.
4. **Only one new method** — `placeCursorAtStart()` is a trivial counterpart to `placeCursorAtEnd()`.
5. **Predictable** — intercepts only at boundaries, preserving the browser's native intra-block and multi-line-block caret movement.

## Risks and Mitigations
| Risk | Impact | Mitigation |
|------|--------|------------|
| `isCursorAtEndOfBlock()` uses ≤2-char tolerance | Arrow handler may fire slightly before visual end in multi-line blocks | Acceptable for now; can tighten tolerance if needed |
| Headings use nested contenteditable | `findEditableElementInBlock()` returns inner `<h1-6>` | Already handled by existing method |
| Code blocks use nested contenteditable | Same as headings | Already handled by `findEditableElementInBlock()` |
| Multi-line blocks (lists, long paragraphs) | Arrow within block should NOT jump to adjacent block | Only intercept when cursor is at boundary (first/last line) |
