# ADR-01: Insert Preceding Block via Enter at Block Start

## Status
Proposed

## Context

Currently, the editor only supports inserting new blocks **after** the current block — by pressing Enter at the end of a block, which calls `addDefaultBlock()` using the DOM `.after()` API. There is no way to insert a block **before** the current block without workarounds (e.g., navigating to the previous block and pressing Enter at its end).

This is a common pain point: when the first block in the editor is a heading or any non-paragraph block, the user has no intuitive way to add content above it. The same applies to any block where the user wants to insert a new block above without leaving the current block.

The proposed solution is to automatically insert a new empty paragraph block **above** the current block when the user presses Enter while the cursor is at the very beginning of the block (position 0, with no text selected).

### Industry Research

This "Enter at beginning inserts before" behavior is a well-established pattern in modern block-based editors:

| Editor | Behavior |
|--------|----------|
| **Notion** | Pressing Enter at the start of a non-empty block inserts a new empty block above |
| **Google Docs** | Enter at the beginning of a paragraph pushes existing content down, effectively creating a preceding line/block |
| **WordPress (Gutenberg)** | Enter at the start of a block creates a new block above; dedicated "Add block before" button also available |
| **Tiptap / ProseMirror** | Enter at position 0 splits the node, creating an empty node before the current content |
| **Editor.js** | Has an explicit "Add block above" action; Enter at start creates a new block before |
| **Confluence** | Enter at the beginning of a block creates a new paragraph above |

This is the dominant pattern across block-based and rich-text editors. Users coming from any of these editors will expect this behavior intuitively.

### Current Code Path

The Enter key flow in `KeyHandler.handleEnterKey()` currently:

1. Checks for triple-backtick code block trigger
2. Delegates to the block's `handleEnterKey()` — if it returns `true`, stops
3. Checks if cursor is at the **end** of the block → calls `addDefaultBlock()` (inserts after)
4. If cursor is **not** at the end → lets the browser handle it (line break / split)

Step 4 is where the gap exists. When the cursor is at the start of a block, the browser inserts a `<br>` or splits inline content, which is incorrect for a block-based editor where each block should be an independent unit.

## Decision

Add "insert preceding block" behavior by detecting cursor-at-start position and inserting a new paragraph block before the current block using the DOM `.before()` API. This requires:

1. **A new cursor detection method**: `isCursorAtStartOfBlock()` on `KeyHandler` (mirrors existing `isCursorAtEndOfBlock()`)
2. **A new insertion method**: `addDefaultBlockBefore()` on `Editor` (mirrors existing `addDefaultBlock()`)
3. **Modified Enter key flow**: Insert a cursor-at-start check in `handleEnterKey()` between steps 2 and 3

### Design

#### Affected Components

| Component | Change |
|-----------|--------|
| `KeyHandler.js` | Add static `isCursorAtStartOfBlock()` method; modify `handleEnterKey()` flow |
| `Editor.js` | Add `addDefaultBlockBefore()` method |

#### New Method: `KeyHandler.isCursorAtStartOfBlock(block, range)`

```javascript
static isCursorAtStartOfBlock(block, range) {
    if (!range || !block || !range.collapsed) {
        return false;
    }
    const preRange = range.cloneRange();
    preRange.selectNodeContents(block);
    preRange.setEnd(range.startContainer, range.startOffset);
    return preRange.toString().length === 0;
}
```

This checks that no text content precedes the cursor within the block. Unlike `isCursorAtEndOfBlock()` which allows a 2-character margin, this must be exact (position 0 only) to avoid false positives when the user simply wants a line break early in the block.

#### New Method: `Editor.addDefaultBlockBefore()`

Mirrors `addDefaultBlock()` but uses `.before()` instead of `.after()`:

```javascript
addDefaultBlockBefore() {
    this._stateMachine.startCreating();
    
    const block = new Block(BlockType.PARAGRAPH);
    const htmlBlock = Parser.html(block);
    
    // Link block instance (same as addDefaultBlock)
    const typedBlock = block.getBlockInstance ? block.getBlockInstance() : block;
    if (typedBlock && typedBlock.element !== undefined) {
        typedBlock.element = htmlBlock;
    }
    this._blockMap.set(htmlBlock, typedBlock);
    
    const blockId = 'block-' + Date.now() + '-' + Math.random().toString(36).substr(2, 9);
    htmlBlock.setAttribute('data-block-id', blockId);
    htmlBlock.setAttribute('data-timestamp', Date.now().toString());
    
    const currentBlock = this.currentBlock;
    if (!currentBlock) {
        this.instance.prepend(htmlBlock);
    } else {
        currentBlock.before(htmlBlock);
    }

    // Keep focus on the CURRENT block (not the new one) — the user's
    // cursor stays where it was; the new empty block appears above.
    this.eventEmitter.emit(EVENTS.BLOCK_CREATED, {
        blockId,
        blockType: BlockType.PARAGRAPH,
        position: Array.from(this.instance.querySelectorAll('.block')).indexOf(htmlBlock),
        timestamp: Date.now()
    }, { source: 'editor.create' });

    requestAnimationFrame(() => {
        this._stateMachine.finishCreating();
    });
    
    return htmlBlock;
}
```

Key difference from `addDefaultBlock()`: focus remains on the current block, not the newly inserted one. The new block is an empty paragraph that appears above — the user continues typing where they are.

#### Modified Enter Key Flow

In `handleEnterKey()`, after block-specific handling (step 2) and before the end-of-block check (step 3), add:

```javascript
// Check if cursor is at the start of the block — insert preceding block
const isAtStart = KeyHandler.isCursorAtStartOfBlock(currentBlock, range);
if (isAtStart && currentBlock.textContent.trim().length > 0) {
    e.preventDefault();
    this.editorInstance.addDefaultBlockBefore();
    return;
}
```

The `textContent.trim().length > 0` guard ensures this only applies to **non-empty** blocks. For empty blocks, pressing Enter should still create a block after (the existing behavior), since inserting before an empty block would be confusing.

### Applicability Per Block Type

| Block Type | Insert-Before Applies? | Notes |
|------------|----------------------|-------|
| **ParagraphBlock** | ✅ Yes | Standard behavior — Enter at start of text inserts paragraph above |
| **H1–H6 (HeadingBlock)** | ✅ Yes | Primary use case — heading as first block needs a way to add content above |
| **QuoteBlock** | ✅ Yes | Same as paragraph — Enter at start inserts paragraph above |
| **DelimiterBlock** | ✅ Yes | Useful to add content above a horizontal rule |
| **ImageBlock** | ⚠️ Special | `handleEnterKey()` returns `true` (always handled). Current behavior creates block after. Could be extended separately if needed. |
| **CodeBlock** | ❌ No | `handleEnterKey()` returns `true` — Enter creates a newline inside the code block. Ctrl+Enter already creates a block after. No change needed. |
| **ListBlocks (UL/OL/Task)** | ❌ No | `handleEnterKey()` returns `true` — list-specific logic handles item creation. Empty-last-item already exits the list. No change needed. |
| **TableBlock** | ❌ No | `handleEnterKey()` returns `true` — Enter adds a table row. No change needed. |

The key insight: blocks whose `handleEnterKey()` returns `true` already fully manage Enter behavior and are **not affected** by this change. The insert-before logic only affects blocks that fall through to the default `handleEnterKey()` flow in `KeyHandler` — specifically paragraphs, headings, quotes, and delimiters.

#### Heading-Specific Consideration

For heading blocks, the editable content is inside an inner `<h1>`–`<h6>` element. The `isCursorAtStartOfBlock()` method should use the editable inner element (not the outer wrapper `<div>`) for accurate cursor position detection. The existing `handleEnterKey()` logic already resolves the correct block container, so this should work naturally if `isCursorAtStartOfBlock()` operates on the same element that `isCursorAtEndOfBlock()` does.

## Alternatives Considered

| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|-------------|
| **Dedicated keyboard shortcut** (e.g., Shift+Enter or Ctrl+Shift+Enter) | Explicit; no ambiguity | Non-discoverable; users won't know the shortcut exists; not standard across editors | Poor UX — users expect Enter at start to work as in Notion/Gutenberg |
| **Floating "+" button above blocks** | Visual affordance; works for all block types including code/table | Requires UI redesign; adds visual clutter; mouse-dependent | Could complement the Enter behavior but shouldn't replace it |
| **Arrow-Up from start of first block** | Intuitive for "first block" problem | Only solves the first-block case; doesn't help mid-document insertion | Too narrow a solution |
| **Split the block at cursor** (ProseMirror-style) | Standard in document-model editors | More complex to implement; requires content splitting and merging; overkill for blocks that are already atomic units | Block architecture uses independent DOM elements, not a document tree — splitting is unnecessarily complex |
| **Always insert before (regardless of content)** | Simpler logic (no empty-block guard) | Confusing on empty blocks — pressing Enter would insert above and leave an empty block with cursor in a weird state | Bad UX edge case |

## Consequences

### Positive
- Users can insert content before any block without workarounds
- Solves the "can't add content above the first block" problem
- Matches behavior users expect from Notion, Gutenberg, Google Docs, and similar editors
- Minimal code change — leverages existing architecture (`addDefaultBlock` pattern, state machine, block map)
- No circular dependency risks — change is confined to `KeyHandler` and `Editor`

### Negative
- Adds a new cursor detection path to the Enter key flow, slightly increasing complexity
- Users accustomed to Enter at the start creating a line break (browser default) may be surprised, though this browser default is already incorrect for a block-based editor

### Neutral
- `addDefaultBlockBefore()` keeps focus on the current block (not the new one), which differs from `addDefaultBlock()` where focus moves to the new block — this is intentional and matches the expected UX, but is a behavioral asymmetry to be aware of
- Blocks that fully handle Enter themselves (code, list, table, image) are completely unaffected
