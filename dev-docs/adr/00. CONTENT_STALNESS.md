# ADR-00: Content Staleness

## Status

**Resolved**

## Problem

Block objects created during `init()` had `_content` set at construction time. `toMarkdown()` and `toHtml()` read from `_content` which never got updated as the user typed. Content lived only in the DOM after initialization, making block-level serialization methods return stale data.

## Decision

Introduce a DOM-aware sync mechanism in the block hierarchy so that `toMarkdown()` and `toHtml()` always reflect the current DOM state.

### Design

1. **`_element` property on `BaseBlock`** â€” Each block can hold a reference to its associated DOM element. When set, the block can read current content directly from the DOM.

2. **`syncFromElement()` method** â€” A virtual method on `BaseBlock` (overridden per block type) that reads the block's current content from its DOM element and updates `_content`, `_html`, and any block-specific properties (e.g., `_language` for code blocks, `_checked` for task lists, `_headers`/`_rows` for tables).

3. **Lazy sync before serialization** â€” `toMarkdown()` and `toHtml()` call `syncFromElement()` first, ensuring `_content` is always current when reading. When no DOM element is attached (headless / test scenarios), `syncFromElement()` is a no-op and the constructor-provided `_content` is used as before.

4. **Editor maintains a `_blockMap`** (`WeakMap<Element, BaseBlock>`) â€” Maps DOM block elements to their typed block instances. Updated in `init()`, `addDefaultBlock()`, `convertBlockType()`, and `createNewBlock()`. Cleared in `destroy()`.

### Block-specific sync implementations

| Block | Sync source |
|-------|-------------|
| `ParagraphBlock` | `textContent` + `innerHTML` (inline formatting preserved via showdown) |
| `HeadingBlock` | Inner `<h1>`â€“`<h6>` element's `textContent` |
| `CodeBlock` | Inner `<code>` element's `textContent` + language from class |
| `QuoteBlock` | Inner `<blockquote>` element's `textContent` |
| `UnorderedListBlock` | `<li>` elements joined by newline |
| `OrderedListBlock` | `<li>` elements joined by newline |
| `TaskListBlock` | `<li>` elements with checkbox state |
| `TableBlock` | `<th>` headers + `<td>` cells from `<tbody>` rows |
| `ImageBlock` | `<img>` element's `src` and `alt` attributes |
| `DelimiterBlock` | No sync needed (constant output `---`) |

## Consequences

- Block objects are no longer stale after initialization â€” `toMarkdown()` / `toHtml()` return current content when attached to the DOM.
- Backward compatible â€” tests that create blocks without DOM elements continue to work because `syncFromElement()` is a no-op when `_element` is null.
- `Editor.getMarkdown()` / `Editor.getHtml()` continue to use their existing DOM-based extraction (via `_blockElementToMarkdown()` / `_blockElementToHtml()`) for robustness. The block-level methods now serve as a parallel, consistent API.
- `Editor.getBlockForElement(element)` provides a public API to look up the block instance for any DOM element.
