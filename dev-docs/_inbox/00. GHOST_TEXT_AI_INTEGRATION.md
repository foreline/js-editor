# Ghost Text Method for AI Integration

## Summary
Introduce a "ghost text" mechanism into the JS Editor that displays AI-generated suggestions as translucent, inline text within the editing surface. Users can accept, partially accept, or dismiss suggestions without disrupting their writing flow — similar to the inline completions seen in GitHub Copilot, Google Docs, and Notion AI.

## Motivation
- AI-assisted writing and code completion are becoming standard expectations in modern editors. Ghost text is the most ergonomic pattern for delivering inline suggestions because it keeps the user in context.
- Integrators embedding JS Editor in their applications would benefit from a first-class API for injecting AI completions, making the editor competitive with AI-native tools.
- End users get a non-intrusive way to receive suggestions — they can ignore them by continuing to type, or accept with a single keystroke (e.g., Tab).
- This opens the door to downstream features: sentence completion, block-level generation, markdown autoformatting suggestions, and smart template insertion.

## Rough Scope
- **New component/module**: A `GhostText` (or `InlineSuggestion`) module responsible for rendering, positioning, and lifecycle of ghost text spans.
- **Rendering**: Ghost text should appear as grayed-out / translucent characters immediately after the cursor position inside the current block's contenteditable element.
- **Keyboard interaction**: Tab (or a configurable key) to accept; Escape to dismiss; continuing to type naturally dismisses or refines the suggestion.
- **Partial accept**: Allow accepting word-by-word (e.g., Ctrl+Right) in addition to full-line accept.
- **Editor API surface**: Expose instance methods like `editor.suggestGhostText(text, options)` and `editor.clearGhostText()` so integrators can feed suggestions from any AI backend.
- **Event hooks**: New events such as `ghosttext:show`, `ghosttext:accept`, `ghosttext:dismiss` to let integrators track interaction.
- **Block compatibility**: Must work across all block types — paragraphs, headings, lists, code blocks, quotes, etc. Special care needed for blocks with nested structures (lists, tables).
- **Parser interaction**: Ghost text must NOT be included in `toMarkdown()` / `toHtml()` exports or in change-tracking timestamps until accepted.
- **Styling**: A dedicated CSS class (e.g., `.ghost-text`) with configurable opacity, color, and font style.
- **No AI backend included**: The editor itself stays backend-agnostic. Ghost text is a UI/UX primitive; the AI provider is the integrator's responsibility.

## Open Questions
- How should ghost text interact with the existing `InlineMarkdownHandler`? If the user types `**` and the ghost text suggests `bold text**`, should markdown triggers still fire?
- Should ghost text support multi-line / multi-block suggestions, or start with single-block only?
- What is the right DOM strategy — insert actual DOM nodes (e.g., `<span class="ghost-text">`) into the contenteditable, or use a positioned overlay? Each has trade-offs for cursor behavior, selection, and screen readers.
- How does ghost text interact with `KeyHandler` priority? Ghost-text keybindings (Tab, Escape) must not conflict with existing block-level key handling (e.g., Tab in code blocks or lists).
- Should there be a debounce/throttle on the API to prevent flickering when suggestions update rapidly?
- Accessibility: How should ghost text be announced to screen readers? `aria-label`, live regions, or hidden from AT entirely?
- Performance: For large documents, how do we ensure ghost text rendering doesn't trigger costly reflows or interfere with the editor's update cycle?
